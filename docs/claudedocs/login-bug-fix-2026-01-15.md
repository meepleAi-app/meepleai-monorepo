## Problem Description

### Symptoms
- `POST /api/v1/auth/login` returned **500 Internal Server Error**
- Browser login form failed with internal error
- API clients unable to authenticate
- Error message: `InvalidJsonRequestBody` during JSON deserialization

### Error Stack Trace
```
Microsoft.AspNetCore.Http.RequestDelegateFactory.Log.InvalidJsonRequestBody
  at RequestDelegateFactory.HandleRequestBodyAndCompileRequestDelegateForJson
  → JSON parsing failed before reaching handler
```

### Impact
- **100% login failure rate** (browser + API)
- Users unable to access authenticated features
- Blocking: user library, chat, dashboard, admin features
- Registration endpoint (`/auth/register`) worked fine

## Root Cause Analysis

### Investigation Process

1. **Compared working vs broken endpoints**:
   - ✅ `/auth/register` → Working (200 OK)
   - ❌ `/auth/login` → Broken (500 Error)

2. **Analyzed payload models**:
   ```csharp
   // RegisterPayload (WORKING)
   internal class RegisterPayload
   {
       public string Email { get; set; } = string.Empty;  // ✅ CORRECT
       public string Password { get; set; } = string.Empty;
   }

   // LoginPayload (BROKEN)
   internal class LoginPayload
   {
       public string Email { get; set; } = default!;      // ❌ WRONG
       public string Password { get; set; } = default!;
   }
   ```

3. **Identified the issue**:
   - `= default!` for `string` type resolves to `null!` (null-forgiving operator)
   - .NET 9 Minimal API JSON deserializer validates non-null constraints
   - Properties initialized with `null!` fail validation during deserialization
   - `= string.Empty` provides valid default value, allowing successful deserialization

### Why RegisterPayload Worked

RegisterPayload used `= string.Empty` from the beginning:
```csharp
// apps/api/src/Api/Models/AuthContracts.cs (line 15-20)
public string Email { get; set; } = string.Empty;     // Valid default
public string Password { get; set; } = string.Empty;  // Valid default
```

This pattern is consistent with .NET 9 nullable reference types + Minimal API requirements.

---

## Solution

### Code Changes

**File**: `apps/api/src/Api/Models/AuthContracts.cs`

**Lines 44, 49**: Changed initializers from `default!` to `string.Empty`

```diff
  internal class LoginPayload
  {
-     public string Email { get; set; } = default!;
+     public string Email { get; set; } = string.Empty;

-     public string Password { get; set; } = default!;
+     public string Password { get; set; } = string.Empty;
  }
```

### Why This Fix Works

1. **Non-null guarantee**: `string.Empty` is a valid non-null string
2. **Deserialization success**: JSON deserializer can instantiate object with valid defaults
3. **Validation preserved**: Endpoint still validates empty strings (lines 87-90 in AuthenticationEndpoints.cs)
4. **Pattern consistency**: Aligns with RegisterPayload pattern

## Prevention Measures

### Code Review Checklist
- [ ] All DTO/Payload classes use `= string.Empty` for required strings
- [ ] Never use `= default!` for non-nullable reference types in payloads
- [ ] Test both registration AND login endpoints after payload changes
- [ ] Run integration tests covering authentication flows

### Pattern to Follow
```csharp
// ✅ CORRECT: Use string.Empty for required strings
public string Email { get; set; } = string.Empty;

// ❌ WRONG: Never use default! for strings in payloads
public string Email { get; set; } = default!;

// ✅ CORRECT: Use null for optional strings
public string? DisplayName { get; set; }
```

### Automated Detection

Add analyzer rule to catch this pattern:
```xml
<!-- Directory.Build.props or .editorconfig -->
<PropertyGroup>
  <WarningsAsErrors>$(WarningsAsErrors);CS8618</WarningsAsErrors>
  <!-- CS8618: Non-nullable property must contain a non-null value -->
</PropertyGroup>
```
