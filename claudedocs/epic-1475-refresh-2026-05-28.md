## Why an umbrella

The pilot game-detail audit revealed three distinct gap categories that each need different handling:

1. **Route implementation gaps** — mockup is ready, stub exists (`return null`), no production code yet.
2. **Conformity gaps on already-merged routes** — the route renders, but a side-by-side mockup audit (like [`#1396`](https://github.com/meepleAi-app/meepleai-monorepo/pull/1396) for `/games/[id]`) hasn't been performed, so silent drift is unbounded.
3. **Missing mockup specifications** — a route exists in code (sometimes with active implementation) but the design source of truth is missing, blocking conformity work entirely.

A single umbrella keeps the cross-category dependency graph visible (e.g. you can't conformity-audit a route whose mockup is missing → mockup gaps must land before audit issues for the same route can move).

## Cross-category dependency rules

1. **C blocks A** for the same route — you can't conformity-audit a route until its mockup exists. So B11/B14/B15 need to land **before** any A-series sibling could start for those routes.
2. **A blocks B** for the same route — you can't conformity-audit `/players/[id]` (B-series) until the impl (A3) lands, because the audit checks `features/player-detail/*` against the mockup.
3. **Tier L routes require Phase 0.5 sub-hook contract** before single-shot dispatch (see [v2 spec §3.4](../docs/for-developers/specs/2026-04-26-v2-design-migration.md#34-phase-05--sub-hook-contract-per-tier-l-routes-only)). This applies to `/discover` and `/knowledge-base/global`.
