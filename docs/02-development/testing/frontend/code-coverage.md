# Frontend Code Coverage Guide

This guide explains how to run code coverage analysis for the MeepleAI frontend (Next.js/React/TypeScript).

## Quick Start

### Run Coverage with HTML Report

```bash
# Run tests with coverage and open HTML report
./tools/run-frontend-coverage.sh --open

# Or manually
cd apps/web
pnpm test:coverage
open coverage/lcov-report/index.html  # View HTML report
```

### Watch Mode

```bash
# Run coverage in watch mode (auto-rerun on file changes)
./tools/run-frontend-coverage.sh --watch
```

## Coverage Script

### Usage

```bash
# Basic usage (90% threshold)
./tools/run-frontend-coverage.sh

# Open HTML report in browser
./tools/run-frontend-coverage.sh --open

# Set custom threshold
./tools/run-frontend-coverage.sh --threshold 85

# Watch mode
./tools/run-frontend-coverage.sh --watch

# Help
./tools/run-frontend-coverage.sh --help
```

### Features

- ✅ Automatic dependency installation check
- ✅ Coverage threshold validation (90% default)
- ✅ HTML report generation (Jest built-in)
- ✅ Coverage summary display (lines, statements, functions, branches)
- ✅ Color-coded console output
- ✅ Watch mode support

### Coverage Files

| File | Location | Format | Purpose |
|------|----------|--------|---------|
| LCOV Report | `apps/web/coverage/lcov.info` | LCOV | Coverage data for CI/Codecov |
| HTML Report | `apps/web/coverage/lcov-report/index.html` | HTML | Human-readable interactive report |
| JSON Summary | `apps/web/coverage/coverage-summary.json` | JSON | Machine-readable summary |
| JSON Detail | `apps/web/coverage/coverage-final.json` | JSON | Detailed coverage data |
| Clover XML | `apps/web/coverage/clover.xml` | XML | Alternative format for tools |

### HTML Report Features

The HTML report generated by Jest includes:

- **Summary Dashboard**: Overall coverage metrics
- **File Browser**: Navigate coverage by directory/file
- **Source View**: Line-by-line coverage highlighting
  - ✅ Green: Covered lines
  - ❌ Red: Uncovered lines
  - 🟡 Yellow: Partially covered (branches)
- **Function Coverage**: See which functions are tested
- **Branch Coverage**: Track conditional coverage
- **Search**: Find specific files/code

## Coverage Thresholds

Current project thresholds (enforced):

| Metric | Threshold | Enforcement |
|--------|-----------|-------------|
| Lines | ≥90% | ✅ Strict |
| Statements | ≥90% | ✅ Strict |
| Functions | ≥90% | ✅ Strict |
| Branches | ≥90% | ✅ Strict |

### Threshold Configuration

Configured in `apps/web/jest.config.js`:

```javascript
coverageThreshold: {
  global: {
    branches: 90,
    functions: 90,
    lines: 90,
    statements: 90,
  },
}
```

### Per-File Thresholds

You can add per-directory/file thresholds:

```javascript
coverageThreshold: {
  global: {
    branches: 90,
    functions: 90,
    lines: 90,
    statements: 90,
  },
  './src/components/': {
    branches: 95,
    functions: 95,
  },
  './src/lib/api/': {
    lines: 85,  // Lower threshold for API client
  },
}
```

## Coverage Collection

### Files Included

Coverage is collected from:

```javascript
collectCoverageFrom: [
  'src/**/*.{js,jsx,ts,tsx}',  // All source files
  '!src/**/*.d.ts',             // Exclude type definitions
  '!src/**/__tests__/fixtures/**',
  '!src/**/__tests__/utils/**',
  '!src/test-utils/**',
  '!src/**/test-utils.{ts,tsx}',
  '!src/**/*.worker.{js,jsx,ts,tsx}',
  '!src/workers/**',
]
```

### Files Excluded

- Type definition files (`*.d.ts`)
- Test fixtures and utilities
- Web workers
- Generated files
- E2E tests (`/e2e/`)

## Understanding Coverage Metrics

### Lines Coverage

Percentage of code lines executed during tests.

```typescript
// Example: 2/3 lines covered (66.67%)
function divide(a: number, b: number): number {
  if (b === 0) throw new Error("Division by zero");  // ✅ Covered
  return a / b;  // ❌ Not covered (test only checks exception)
}
```

### Statements Coverage

Similar to lines but counts individual statements.

```typescript
// Example: 3/4 statements covered (75%)
function getUserName(user: User | null): string {
  if (!user) return "Guest";  // ✅ Covered
  const name = user.name;     // ✅ Covered
  return name.toUpperCase();  // ❌ Not covered
}
```

### Functions Coverage

Percentage of functions called during tests.

```typescript
// Example: 1/2 functions covered (50%)
export function add(a: number, b: number) {
  return a + b;  // ✅ Covered (has test)
}

export function multiply(a: number, b: number) {
  return a * b;  // ❌ Not covered (no test)
}
```

### Branches Coverage

Percentage of decision paths executed.

```typescript
// Example: 3/4 branches covered (75%)
function getStatus(score: number): string {
  if (score >= 90) return "Excellent";  // ✅ Covered
  if (score >= 70) return "Good";       // ✅ Covered
  if (score >= 50) return "Pass";       // ✅ Covered
  return "Fail";                        // ❌ Not covered
}
```

## Best Practices

### 1. Focus on Critical Code

Prioritize testing and coverage for:
- ✅ Business logic components
- ✅ State management (hooks, stores)
- ✅ API integration code
- ✅ Form validation
- ✅ Complex calculations
- ✅ Error handling

Lower priority:
- ⚠️ Simple presentational components
- ⚠️ Type definitions
- ⚠️ Configuration files
- ⚠️ Storybook stories

### 2. Test Behavior, Not Implementation

```typescript
// ❌ Bad: Testing implementation details
test('LoginForm sets isSubmitting to true', () => {
  const { result } = renderHook(() => useLoginForm());
  act(() => result.current.setIsSubmitting(true));
  expect(result.current.isSubmitting).toBe(true);
});

// ✅ Good: Testing user behavior
test('LoginForm shows loading state when submitting', async () => {
  render(<LoginForm />);
  const submitButton = screen.getByRole('button', { name: /log in/i });

  fireEvent.click(submitButton);

  expect(submitButton).toBeDisabled();
  expect(screen.getByText(/logging in/i)).toBeInTheDocument();
});
```

### 3. Use Coverage to Find Gaps

Use coverage reports to identify:
- ❌ Untested edge cases
- ❌ Missing error handling tests
- ❌ Uncovered validation logic
- ❌ Untested user interactions

### 4. Don't Chase 100%

High coverage is good, but:
- 90%+ is excellent for most code
- Some code is hard/expensive to test (worth excluding)
- Focus on valuable tests, not coverage numbers
- Quality > Quantity

### 5. Exclude Intentionally

Use Istanbul ignore comments for untestable code:

```typescript
/* istanbul ignore next */
if (process.env.NODE_ENV === 'development') {
  console.log('Debug info:', data);
}

/* istanbul ignore if */
if (typeof window === 'undefined') {
  return null;  // Server-side only
}
```

## Troubleshooting

### Coverage Below Threshold

**Problem**: Tests pass but coverage is below threshold

**Solutions**:

1. **Identify uncovered files**:
   ```bash
   ./tools/run-frontend-coverage.sh --open
   ```
   Look for red/yellow in the HTML report

2. **Check the console output**:
   Jest prints which files don't meet the threshold

3. **Add missing tests**:
   - Focus on business logic first
   - Test error cases
   - Cover edge cases
   - Test user interactions

4. **Consider if code is testable**:
   - Is it too complex? Refactor.
   - Is it tightly coupled? Decouple.
   - Can it be simplified?

### Tests Hanging

**Problem**: Tests hang during execution

**Solutions**:

1. Check for:
   - Infinite loops
   - Unresolved promises
   - Missing `await` in async tests
   - Timer mocks not advancing

2. Use Jest's timeout:
   ```typescript
   test('async operation', async () => {
     // ...
   }, 10000);  // 10 second timeout
   ```

3. Run with verbose output:
   ```bash
   pnpm test:coverage -- --verbose
   ```

### Out of Memory Errors

**Problem**: Jest runs out of memory on large codebases

**Solutions**:

1. Increase Node.js memory:
   ```bash
   NODE_OPTIONS=--max_old_space_size=4096 pnpm test:coverage
   ```

2. Run tests in sequence:
   ```bash
   pnpm test:coverage -- --runInBand
   ```

3. Limit workers:
   ```bash
   pnpm test:coverage -- --maxWorkers=2
   ```

### Coverage Report Not Generated

**Problem**: Tests run but no coverage report appears

**Solutions**:

1. Ensure `--coverage` flag is used:
   ```bash
   pnpm test -- --coverage
   ```

2. Check Jest configuration:
   - Verify `coverageDirectory` is set
   - Check `collectCoverageFrom` patterns

3. Clean and retry:
   ```bash
   rm -rf coverage/
   pnpm test:coverage
   ```

### HTML Report Not Opening

**Problem**: `--open` flag doesn't open browser

**Solutions**:

1. Open manually:
   ```bash
   # macOS
   open apps/web/coverage/lcov-report/index.html

   # Linux
   xdg-open apps/web/coverage/lcov-report/index.html

   # Windows
   start apps/web/coverage/lcov-report/index.html
   ```

2. Check file exists:
   ```bash
   ls -la apps/web/coverage/lcov-report/index.html
   ```

## Coverage Tools

### Jest

Primary testing and coverage tool for frontend.

**Features**:
- ✅ Built-in coverage collection (Istanbul)
- ✅ Multiple report formats (HTML, LCOV, JSON, Clover)
- ✅ Threshold enforcement
- ✅ Watch mode
- ✅ Fast parallel execution

**Documentation**: https://jestjs.io/docs/configuration#coverage

### Istanbul (NYC)

Coverage instrumentation engine used by Jest.

**Features**:
- ✅ Accurate code instrumentation
- ✅ Branch coverage tracking
- ✅ Ignore annotations
- ✅ Source map support

**Documentation**: https://istanbul.js.org/

### Codecov

Cloud service for coverage tracking.

**Features**:
- 📊 Coverage dashboards
- 📈 Trend analysis
- 💬 PR comments with coverage diff
- 🚨 Alerts for coverage drops
- 🎯 File-level coverage tracking
- 🔀 Compare branches

**Dashboard**: https://codecov.io/gh/DegrassiAaron/meepleai-monorepo

## Advanced Usage

### Coverage for Specific Tests

```bash
# Run coverage for specific test file
pnpm test:coverage -- path/to/test.spec.ts

# Run coverage for tests matching pattern
pnpm test:coverage -- --testNamePattern="LoginForm"

# Run coverage for specific directory
pnpm test:coverage -- src/components/
```

### Custom Coverage Reporters

Add custom reporters in `jest.config.js`:

```javascript
coverageReporters: [
  'text',          // Console output
  'text-summary',  // Summary in console
  'html',          // HTML report
  'lcov',          // LCOV for Codecov
  'json',          // JSON format
  'cobertura',     // Cobertura XML
],
```

### Coverage with Watch Mode

```bash
# Run coverage in watch mode
./tools/run-frontend-coverage.sh --watch

# Or manually
pnpm test:coverage -- --watch

# Watch specific files
pnpm test:coverage -- --watch --testPathPattern="components"
```

### Combine with Other Flags

```bash
# Coverage with verbose output
pnpm test:coverage -- --verbose

# Coverage with test name pattern
pnpm test:coverage -- --testNamePattern="should render"

# Coverage for changed files only
pnpm test:coverage -- --onlyChanged

# Coverage with bail (stop on first failure)
pnpm test:coverage -- --bail
```

```json
{
```
