# Conformity Baseline Single-Screen — Design Spec

**Date**: 2026-05-18
**Issue**: #1269 (conformity-debt, expires 2026-06-17)
**Umbrella**: #1066 (WS-C conformity framework), #1023 (DS de-versioning)
**Status**: brainstormed — ready for plan
**Authors**: Aaron + Claude

---

## Problem

The conformity gate (`visual-regression-conformity.yml`) compares **live Next.js routes** against **mockup baseline PNGs** generated by `bootstrap-mockup-baselines.yml`. The bootstrap workflow captures `fullPage` of each `admin-mockups/design_files/*.html`, but these mockups are **multi-screen design catalogs** — they expose every state variant of a route stacked vertically (default, loading, empty, error, mobile + desktop frames).

Result: baseline PNGs are 4000–12000px tall, while live routes render a single screen of 900–1800px. Diff is always ~80% pixels different — independent of how mockup-faithful the implementation is.

**Empirical evidence** (PR #1267, run 26030339270, 2026-05-18):

| Route | Baseline (multi-state catalog) | Live route render | Diff ratio |
|-------|--------------------------------|-------------------|------------|
| `/library` | 1280×9669 | 1280×1752 | 0.20 (5.5× height gap) |
| `/library/{gameId}` | 1280×4026 | 1280×907 | 0.21 (4.4× height gap) |
| `/players/{id}` | 1280×11989 | — (no runbook) | crash |
| `/game-nights` | 1280×12320 | — (no runbook) | crash |

This is a **structural bug in the WS-C framework** (introduced PR #1117, #1234), not a Stage 3 cluster regression. PR #1267 waiver rationale ("Stage 3 ongoing") misdiagnosed the root cause.

The 4 routes are tracked under conformity-debt issue **#1269 expiring 2026-06-17**. Without remediation, PRs to `main-staging`/`main` will be blocked by the debt gate.

## Decisions (brainstorming 2026-05-18)

1. **Baseline scope**: Solo `default` state per ogni route+viewport (8 baselines totali). Multi-state coverage delegata a `e2e/v2-states/*.spec.ts` (axe + functional, già esistente).
2. **Marker location**: Nel mockup HTML come `data-conformity-screen="default-{viewport}"`. Self-contained — il mockup descrive le proprie scene.
3. **Fallback policy**: Hard-fail. Mockup senza marker per il viewport corrente → bootstrap.spec.ts errore esplicito.
4. **Naming convention**: `default-desktop` + `default-mobile` — due marker per mockup (i mockup espongono già entrambe le view).
5. **Bootstrap impl**: Modify in-place `bootstrap.spec.ts` → `expect(page.locator('[data-conformity-screen="default-{viewport}"]')).toHaveScreenshot(...)`. Hard-fail se locator vuoto.
6. **Rollout**: PR #1 (code), workflow dispatch post-merge, PR #2 (auto-generated baseline). Separazione code/data; baseline rigenerate su CI Ubuntu deterministic.

### Data flow

```
Mockup HTML (admin-mockups/design_files/*.html/.jsx)
    └── React/HTML renders → includes <section data-conformity-screen="default-desktop">
                                                                   ← marker injection (this spec)

Bootstrap workflow (push on admin-mockups/** OR workflow_dispatch)
    └── bootstrap.spec.ts: page.goto(`/${mockup}`)
                          → page.locator('[data-conformity-screen="default-{viewport}"]').toHaveScreenshot()
                          → screenshot saved to __mockup__/{route.id}.{viewport}.png
                          → auto-PR with new baselines

Conformity gate (push/PR on routes/trigger paths)
    └── conformity.spec.ts: page.goto(livePath)
                           → page.toHaveScreenshot() compare vs __mockup__/{route.id}.{viewport}.png
                           → GREEN if matched, FAIL with diff triplet artifact
```

### Marker contract

```html
<!-- Mockup HTML/JSX exposes one or more design scenes; mark the one matching the live route -->
<div data-conformity-screen="default-desktop">
  <!-- The exact content the live route renders at 1280×N viewport -->
</div>
<div data-conformity-screen="default-mobile">
  <!-- The exact content the live route renders at 375×N viewport -->
</div>
```

**Rules:**
- Exactly **one** element per `data-conformity-screen` value per mockup (multiple matches → bootstrap fails).
- The marked element MUST be in normal document flow (no `position:absolute/fixed` on the wrapper, no `transform:scale(...)`) so `boundingBox()` returns full-content dimensions.
- The marked element MUST NOT be hidden via `display:none` / `visibility:hidden` / `opacity:0`.
- Values are case-sensitive lowercase: `default-desktop`, `default-mobile`. Future extensions (`empty-desktop`, etc.) are out of scope for this spec (see Architecture note `Marker naming v1 = default-{viewport}` below).
- v1 contract: only `default-desktop` and `default-mobile` supported. Extension to multi-state (e.g. `empty-desktop`) tracked as Out of Scope §2; would require ownership.json `screens: [...]` field.

### Edge Cases & Test Matrix

| Mockup type | Render style | Marker mount timing | bootstrap.spec.ts behavior |
|-------------|--------------|---------------------|----------------------------|
| React-bootstrapped (sp4-*.jsx) | JSX 8-12s compile | Post-React mount in #root | `waitForMockupReady` React branch (esistente) → poi `page.locator(marker).waitFor()` con timeout 30s |
| Pure-CSS HTML (nanolith-runthrough) | Immediate | DOM-ready | `waitForMockupReady` pure-CSS branch (esistente) → marker già nel DOM iniziale |
| Mockup con shadow DOM | N/A | N/A | **OUT OF SCOPE v1**. Mockup attuali no shadow DOM. Docs explicit exclusion: marker MUST be light-DOM. |
| Mockup con iframe | N/A | N/A | **OUT OF SCOPE v1**. Marker MUST be in main document, NOT in nested frames. |
| Mockup con marker dentro scrollable container | Visible after scroll | Mounted but offscreen | Playwright `scrollIntoViewIfNeeded()` auto-handled da `toHaveScreenshot()` |
| Marker con `display:none` o `visibility:hidden` | Hidden | Mounted but invisible | `boundingBox()` returns null → Playwright throws (acceptable hard-fail, documented in rules) |

### Bootstrap test (after modification)

```typescript
test(`mockup baseline: ${route.id} (${route.mockup})`, async ({ page }, testInfo) => {
  const viewport = testInfo.project.name.includes('mobile') ? 'mobile' : 'desktop';
  const selectorValue = `default-${viewport}`;
  const selector = `[data-conformity-screen="${selectorValue}"]`;

  await page.goto(`/${route.mockup}`, { waitUntil: 'networkidle' });
  await waitForMockupReady(page);

  const target = page.locator(selector);
  const count = await target.count();
  if (count === 0) {
    throw new Error(
      `Mockup ${route.mockup} missing required marker. ` +
      `Add data-conformity-screen="${selectorValue}" to the element that ` +
      `represents the canonical default ${viewport} screen.`
    );
  }
  if (count > 1) {
    throw new Error(
      `Mockup ${route.mockup} has ${count} elements with data-conformity-screen="${selectorValue}". ` +
      `Exactly one is required.`
    );
  }

  await expect(target).toHaveScreenshot(`${route.id}.png`);
});
```

## Out of Scope (Open Issues for future work)

1. **Visual-test fixture for game-nights index** — `e2e/v2-states/game-nights-index.spec.ts` uses runtime API stubs; for conformity we need build-time `NEXT_PUBLIC_VISUAL_TEST_FIXTURE_ENABLED` short-circuit in `_content.tsx` (orchestrator). If the runbook readySelector is sufficient and the orchestrator already has a deterministic empty render, AC-4 can pass without this. Verify during implementation; if needed, defer to **follow-up issue** with effort ~3-5h.

2. **Multi-state baselines** (default + empty + error per route) — explicit YAGNI decision. State coverage stays in `e2e/v2-states/*.spec.ts`. Reopen as design discussion in #1023 if needed.

3. **Validator for marker policy** in `pnpm verify:mockup-pins` — prevent new mockups from being added without marker. Track as follow-up after first new mockup creation post-merge.

4. **Reuse of marker pattern for `bootstrap-single-screen.spec.ts`** outside conformity (e.g. visual regression of cards in isolation) — out of scope.
