### Performance Issue Identified

**Root Cause**: Redis offline → Fail-open strategy

Each Redis operation (rate limit, session cache) attempts connection and times out after **5000ms**:
```
StackExchange.Redis.RedisConnectionException:
  The message timed out in the backlog attempting to send
  because no connection became available (5000ms)
```

**Impact**:
- Each test makes ~3-4 Redis operations (rate limit + session cache)
- Each operation waits 5s before fail-open
- Test time: **~30s** (with Redis offline) vs. **~10s** (with Redis running)
- **Total overhead**: ~2-3x slower

**Why Tests Still Pass**:
The application uses **fail-open** strategy for Redis:
- Rate limit: Allows request if Redis unavailable
- Session cache: Falls back to database if Redis unavailable

This is **correct behavior** for production resilience, but slows tests significantly.

## Recommendations

### For Local Development

**Start Redis before running tests** to avoid timeout delays:

```bash
# Option 1: Docker Compose (recommended)
cd infra
docker compose up redis -d

# Option 2: Standalone Docker
docker run -d -p 6379:6379 redis:7-alpine

# Then run tests (now ~3 minutes instead of ~10 minutes)
cd apps/api
dotnet test --filter "FullyQualifiedName~QualityTrackingIntegrationTests"
```

### For CI

Tests already configured to use service container via environment variable:
```yaml
# .github/workflows/ci.yml

    ports:

```
