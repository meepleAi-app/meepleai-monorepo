## Closes #1188 — load MaD sanitizer pack via `--additional-packs` env var

Phase 1.4 of #1181 originally referenced the MaD pack via the `packs:` key in `codeql-config.yml`:

```yaml
packs:
  - .github/codeql/extensions/meepleai-sanitizers
```

Post-merge CodeQL run on `main-dev` revealed CodeQL CLI 2.25.4 silently rejects path-based references in that key (it parses each entry as a `scope/name@version:path` pack spec):

```
WARNING: Invalid CodeQL pack specification:
'.github/codeql/extensions/meepleai-sanitizers'. Ignoring.
```

Local validation (T2) had used `gh codeql resolve extensions --additional-packs <path>` which goes through a different resolution path and masked the issue.

### What this PR does

1. **Removes** the broken `packs:` entry from `.github/codeql/codeql-config.yml` — keeping it only generated the misleading warning.
2. **Adds** `CODEQL_ACTION_EXTRA_OPTIONS` env var on the `init@v4` step in `.github/workflows/security-scan.yml`. The env var is the documented escape hatch — its value is parsed JSON whose nested keys map to CodeQL CLI commands and whose array values are appended as extra flags. Both `database init` and `database analyze` get `--additional-packs=.github/codeql/extensions` so the pack is discoverable at config-parsing time AND when extensions auto-apply during analysis:
   ```yaml
   env:
     CODEQL_ACTION_EXTRA_OPTIONS: |
       {
         "database": {
           "init": ["--additional-packs=.github/codeql/extensions"],
           "analyze": ["--additional-packs=.github/codeql/extensions"]
         }
       }
   ```

### What this PR does NOT do

Does **not** remove the `cs/log-forging` blanket exclude restored by PR #1187. That removal is a separate follow-up PR after this one merges and the next CodeQL run on `main-dev` empirically confirms the pack is loading and recognizing call sites. Two PRs because the failure mode of "exclude removed but pack still not loading" produces a visible regression of 84 alerts — exactly what happened the first time.
