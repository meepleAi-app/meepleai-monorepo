# Environment Configuration Files

Questa directory contiene file di esempio per configurare i 3 ambienti di MeepleAI.

## 📁 File Disponibili

| File | Ambiente | Uso | Stato |
|------|----------|-----|-------|
| `.env.development.example` | Development (locale) | Sviluppo attivo | ✅ Template pronto |
| `.env.staging.example` | Staging (pre-produzione) | QA testing, demo | ⚠️ Richiede configurazione |
| `.env.production.example` | Production (live) | Utenti reali | 🔒 Richiede secrets |

## 🚀 Quick Start

### Development (Locale)

```bash
# 1. Copia file esempio
cp .env.development.example .env.development

# 2. (Opzionale) Modifica valori locali
# vim .env.development

# 3. Avvia stack con Docker Compose
cd infra
docker compose --env-file ../.env.development up -d

# 4. Verifica
curl http://localhost:8080/health
```

### Staging

```bash
# 1. Copia file esempio
cp .env.staging.example .env.staging

# 2. Sostituisci TUTTI i <REPLACE_*> placeholders
vim .env.staging
# Esempio:
#   <REPLACE_WITH_DB_HOST> → staging-db.internal
#   <REPLACE_WITH_STRONG_PASSWORD> → GeneraP@ssw0rdForte!

# 3. Deploy su infrastruttura cloud
cd infra
docker compose -f compose.staging.yml --env-file ../.env.staging up -d

# 4. Verifica
curl https://staging.meepleai.dev/health
```

### Production

```bash
# 1. Copia file esempio
cp .env.production.example .env.production

# 2. Sostituisci TUTTI i <REPLACE_*> placeholders
# ⚠️  NON editare direttamente - usa secret management
# Esempio: AWS Secrets Manager, Azure Key Vault, HashiCorp Vault

# 3. Configura Docker Secrets
cd infra/secrets
./init-secrets.sh --environment production

# 4. Deploy
cd infra
docker compose -f compose.prod.yml --env-file ../.env.production up -d

# 5. Verifica
curl https://api.meepleai.dev/health
```

## 📋 Gestione File .env

### ✅ Cosa FARE

```bash
# Usa file .local per override personali (gitignored)
cp .env.development .env.development.local
# Modifica solo .local

# Version control dei file .example
git add .env.development.example
git commit -m "docs: update development config example"

# Backup sicuro dei file production (encrypted)
gpg --encrypt --recipient ops@meepleai.dev .env.production
```

### ❌ Cosa NON FARE

```bash
# MAI committare file .env con valori reali
git add .env.production  # ❌ PERICOLOSO
git add .env.staging     # ❌ PERICOLOSO

# MAI condividere credenziali via email/Slack
cat .env.production | mail ops@example.com  # ❌ INSICURO

# MAI usare credenziali production in development
cp .env.production .env.development  # ❌ MALE
```

## 🔐 Sicurezza

### File .gitignore

Il repository include automaticamente nel `.gitignore`:

```gitignore
# Environment files (DO NOT COMMIT)
.env
.env.local
.env.*.local
.env.development
.env.staging
.env.production

# Committabili (esempi)
.env.*.example
```

### Secrets Management

Per **staging** e **production**, usare **Docker Secrets**:

```bash
# Directory secrets
infra/secrets/
├── dev/                  # Locale (semplici)
├── staging/              # Staging (forti)
└── prod/                 # Production (ultra-forti)

# Inizializzazione
cd infra/secrets
./init-secrets.sh --environment production

# Rotazione (ogni 90 giorni in prod)
./rotate-secret.sh postgres-password --environment production
```

## 🔍 Troubleshooting

### Problema: Variabile non trovata

```bash
# Verifica file .env caricato
docker compose config | grep POSTGRES_USER

# Check merge di file
docker compose --env-file .env.development config
```

### Problema: Secret non montato

```bash
# Verifica secrets in container
docker compose exec api ls -la /run/secrets/

# Check contenuto secret
docker compose exec api cat /run/secrets/postgres-password
```

### Problema: Override non funziona

```bash
# Priority: .env.local > .env > .env.example
# Crea .env.development.local
cp .env.development .env.development.local
# Modifica solo variabili da sovrascrivere
```

## 💡 Best Practices

1. **Usa .local per personalizzazioni** - Mai modificare .env.development direttamente
2. **Documenta cambiamenti** - Aggiorna .example quando aggiungi variabili
3. **Valida prima di committare** - Controlla che .example non contenga secrets
4. **Testa in staging** - Deploy a staging PRIMA di production
5. **Backup production** - Backup encrypted di .env.production in vault sicuro

## ⚙️ File .env vs Docker Compose

### Approccio 1: File .env in root

```bash
# .env in root del progetto
.env.development

# Docker Compose carica automaticamente
cd infra
docker compose up -d
```

### Approccio 2: File .env specifico

```bash
# File .env in posizione custom
.env.staging

# Specificare esplicitamente
docker compose --env-file ../.env.staging up -d
```

### Approccio 3: Docker Secrets (production)

```bash
# Secrets in infra/secrets/prod/
postgres-password.txt
openrouter-api-key.txt

# Docker Compose li monta automaticamente
# Nessun .env file necessario per secrets
```
