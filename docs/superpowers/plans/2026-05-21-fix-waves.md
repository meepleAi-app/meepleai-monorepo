### Lessons learned — audit tool limitations exposed

Three categories of residual false positives surfaced during the waves; they are documented in `audit-report-final.md` and should inform any successor toolchain work:

1. **Nav findings on layout/page-level destinations** — `nav_dimension.py` scans components listed in `v2-migration-matrix.md`, not the route's `page.tsx` or `layout.tsx`. Fixes added to a `PageHeader` (with `parentHref`) or to a page-level `<Link>` are invisible. Workaround used: SKIP_FP with rationale "back-link exists in layout.tsx".
2. **Structural `h1/h2` missing in sub-components** — audit compares each sub-component against the full-page mockup, so sub-components legitimately starting at `h3` are flagged. Adding an extra `h1` would violate WCAG (single `h1` per page). Workaround: SKIP_FP with rationale "h1 owned by parent page".
3. **Nav mapping heuristic bugs** — some mockup destinations (e.g. `librogame-runthrough-game-onboarding.html`) map to routes that the live app exposes at different paths (e.g. `/library/games/[id]?tab=…`). Workaround: SKIP_FP with rationale "audit mapping bug".

Successor work (not part of this plan): improve `scripts/v2_audit/nav_dimension.py` to also scan `page.tsx` / `layout.tsx`, and teach the structural dimension about parent-page header context. Tracked informally — open a dedicated issue if the next audit wave re-surfaces the same categories.

## Per-Route Workflow Template

All 10 route tasks follow this identical workflow. The TEMPLATE below is the contract; Tasks 1-10 specify only the issue number, route, finding count, and recommended implementer model.

### TEMPLATE: Single-route fix workflow

**Stage A — Implementer subagent**

Dispatch using `Agent` tool, `subagent_type=general-purpose`, model per task spec. Use the prompt template at the END of this plan (Section "Implementer Prompt Template").

The implementer:
1. Checks out `main-dev`, pulls, creates `feature/audit-fix-<slug>`.
2. Reads issue body via `gh issue view <num> --json title,body`.
3. For each finding: identifies target file, decides FIX/SKIP_FP/DEFER, applies fix.
4. Local verification: `pnpm lint`, `pnpm typecheck`, `pnpm test --run --related`, partial `python -m scripts.v2_audit.run`.
5. Commits, pushes, opens PR with `Closes #NNNN`.
6. Reports stats + PR URL.

**Stage B — Spec compliance reviewer**

Dispatch using `Agent` tool, model `haiku`. Use prompt template at END of plan (Section "Spec Reviewer Prompt Template").

The reviewer:
1. Reads PR diff + body.
2. Verifies every issue finding has decision FIX/SKIP_FP/DEFER documented.
3. Spot-checks 3 random FIX findings: change coherent with mockup intent.
4. Confirms `Closes #NNNN` present.
5. Confirms no out-of-scope file modifications.
6. Returns ✅ Approved / ❌ Issues list.

**Stage C — Code quality reviewer**

Dispatch using `Agent` tool, model `haiku`. Use prompt template at END of plan (Section "Code Quality Reviewer Prompt Template").

The reviewer:
1. Runs ESLint scope check: `cd apps/web && pnpm lint --filter <files>`.
2. Runs typecheck: `pnpm typecheck`.
3. Verifies no new hardcoded color utilities, no `eslint-disable` directives.
4. Verifies tests pass for affected files.
5. Returns ✅ Approved / ❌ Issues list.

**Stage D — Controller (no subagent)**

Manual steps:
1. Wait CI green (poll `gh pr checks <PR>`).
2. Merge: `gh pr merge <PR> --squash --delete-branch`.
3. Switch to main-dev, pull: `git checkout main-dev && git pull --ff-only`.
4. Re-audit: `python -m scripts.v2_audit.run`.
5. Re-create-issues (idempotent): `python -m scripts.v2_audit.create_issues`.
6. Check if issue auto-closed (via `Closes #NNNN`).
   - If still open AND 0 findings remain in the re-audit report:
     `gh issue close <num> --comment "All findings resolved by PR #X (post-merge re-audit confirmed 0 remaining)"`.
   - If still open AND findings remain because of DEFER decisions in the PR:
     `gh issue edit <num> --add-label "audit-deferred"` (do NOT close).
     Verify the sub-issue(s) referenced in the PR body exist and are open.
   - If still open AND findings remain because of SKIP_FP decisions:
     `gh issue close <num> --comment "Remaining findings classified as false positives by PR #X (see PR body Decisions table for rationale)"`.
7. Cleanup local: `git branch -D feature/audit-fix-<slug>`.
8. Commit refreshed audit-report + snapshot:
   `git add docs/superpowers/specs/audit-report.md docs/superpowers/specs/audit-input-validated.json && git commit -m "chore(v2-audit): post-merge refresh after PR #X" && git push`.

# Implementer Prompt Template

```
You are the implementer for V2 audit fix wave: route <ROUTE> (issue #<ISSUE_NUM>, <FINDING_COUNT> findings).

Working dir: D:/Repositories/meepleai-monorepo-main. Parent branch: main-dev (already up to date with origin).

## Steps

1. Create branch:
   git checkout main-dev && git pull --ff-only
   git checkout -b feature/audit-fix-<BRANCH_SLUG>
   git config branch.feature/audit-fix-<BRANCH_SLUG>.parent main-dev

2. Read issue body:
   gh issue view <ISSUE_NUM> --json title,body
   Parse the markdown body. The structure is:
   - "### Critical (N)" section with checkbox items
   - "### Important (N)" section with checkbox items
   Each item has: `<component(s)> (<dimension>): <description>`
   Plus evidence sub-bullets: `expected_route`, `mockup_dest`, `component_hrefs`, `token`, `missing_field`, etc.

3. For each finding (Critical FIRST, then Important):
   a. Identify target file(s):
      - For NAV findings, the `component` field lists MULTIPLE files. To pick the right one, FIRST read `docs/superpowers/specs/nav-map.md` (or grep it for the destination route) to understand the navigation context: is the link a top-level sidebar item (lives in layout/sidebar), a header/breadcrumb (lives in layout/header), or a card action (lives in the feature card)?
        - Sidebar/top-nav Links → `apps/web/src/app/(authenticated)/layout.tsx`, or `apps/web/src/components/layout/Sidebar.tsx` if it exists
        - Header/topbar Links → `apps/web/src/components/layout/Header.tsx`, or the page-level `page.tsx` if no header component
        - Card/list/CTA Links → the most generic feature card in the issue's component list (e.g., `GameCard.tsx`, `PlayerCard.tsx`)
        - If still unclear after consulting nav-map: pick the LAYOUT component (most likely to be the right ownership) and document the choice in the PR Decisions table rationale column.
      - For STRUCTURAL findings, the component is single — open that file
      - For TOKENS findings, the component is single
      - For PROPS findings, the component is single
   b. Open the file with Read tool, locate the section needing modification
   c. Decide: FIX, SKIP_FP, or DEFER
      - FIX: apply the change
      - SKIP_FP: confirm via inspection the finding is wrong (e.g., mockup link is decorative, or component is correctly a subset). Note why.
      - DEFER: complex refactor needed. Open a sub-issue:
        gh issue create --title "<scoped sub-issue title>" --body "Deferred from #<ISSUE_NUM>: <description and rationale>" --label "audit-deferred"
        Record the new issue number for the PR body.
   d. Apply fix patterns:
```
