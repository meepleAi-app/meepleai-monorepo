## What Are Anonymous Volumes?

### Definition

**Anonymous volumes** are Docker volumes created **without explicit names**, identified only by a 64-character hexadecimal hash.

```bash
# Named volume (✅ Good)
DRIVER    VOLUME NAME
local     meepleai_postgres_data

# Anonymous volume (❌ Problematic)
DRIVER    VOLUME NAME
local     7a3d8c9b1e2f48a5c3d1e9f0b4a6c8d2e1f0a9b8c7d6e5f4a3b2c1d0e9f8a7b6
```

### Visual Comparison

```
┌─────────────────────────────────────────────────────────────────┐
│                        NAMED VOLUME                              │
├─────────────────────────────────────────────────────────────────┤
│  Name: meepleai_postgres_data                                   │
│  Purpose: PostgreSQL database storage                           │
│  Created: Explicitly in docker-compose.yml                      │
│  Lifecycle: Persists until explicitly removed                   │
│  Visibility: Easy to identify and manage                        │
│  Backup: Simple with clear name                                 │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                      ANONYMOUS VOLUME                            │
├─────────────────────────────────────────────────────────────────┤
│  Name: a7f3d8c9b1e2f48a5c3d1e9f0b4a6c8d2                        │
│  Purpose: ??? (unknown without inspection)                      │
│  Created: Auto-generated by Docker                              │
│  Lifecycle: Can be removed with docker compose down -v          │
│  Visibility: Hash makes identification hard                     │
│  Backup: Difficult to automate                                  │
└─────────────────────────────────────────────────────────────────┘
```

## How They Are Created

### 1. Dockerfile VOLUME Directive

**Scenario**: Base image declares VOLUME, but docker-compose doesn't override

```dockerfile
# postgres:16 Dockerfile (simplified)
FROM debian:bullseye-slim
VOLUME /var/lib/postgresql/data  # Creates anonymous volume if not overridden
```

```yaml
# docker-compose.yml - Creates ANONYMOUS volume
services:
  postgres:
    image: postgres:16
    # No volumes section = Docker uses VOLUME from Dockerfile
    # Result: Anonymous volume created
```

**Fix**:
```yaml
# docker-compose.yml - Override with named volume
services:
  postgres:
    image: postgres:16
    volumes:
      - postgres_data:/var/lib/postgresql/data  # Named volume

volumes:
  postgres_data:  # Explicit declaration
```

### 2. Unnamed Volume Mount

**Scenario**: Volume path without name

```yaml
# docker-compose.yml - Creates ANONYMOUS volume
services:
  app:
    volumes:
      - /app/data  # No name = anonymous!
```

**Fix**:
```yaml
# docker-compose.yml - Use named volume
services:
  app:
    volumes:
      - app_data:/app/data  # Named volume

volumes:
  app_data:
```

### 3. CLI Volume Flag

**Scenario**: `docker run` without volume name

```bash
# Creates ANONYMOUS volume
docker run -v /data postgres:16

# Creates NAMED volume
docker run -v postgres_data:/var/lib/postgresql/data postgres:16
```

## Problems with Anonymous Volumes

### 1. Difficult to Identify

```bash
# Which volume belongs to which service?
docker volume ls
# local     a7f3d8c9b1e2...  # Postgres? Redis? Unknown!
# local     2c1b8a9d3e4f...  # What is this?
# local     5f4a3b2c1d0e...  # No idea!
```

### 2. Orphaned Volumes Accumulate

```bash
# After rebuilding containers multiple times
docker volume ls | wc -l
# 47 volumes! (should be ~10)

# Most are orphaned anonymous volumes
docker volume ls --filter "dangling=true" | wc -l
# 37 dangling volumes wasting disk space!
```

### 3. Backup Nightmares

```bash
# Named volume: Clear purpose
docker run --rm \
  -v meepleai_postgres_data:/source:ro \
  -v /backups:/backup \
  alpine tar czf /backup/postgres.tar.gz -C /source .

# Anonymous volume: Which one to backup?
docker run --rm \
  -v a7f3d8c9b1e2...:/source:ro \  # Is this the right one?!
  -v /backups:/backup \
  alpine tar czf /backup/unknown.tar.gz -C /source .
```

### 4. Accidental Data Loss

```bash
# User thinks: "I'll just clean up unused volumes"
docker volume prune -f

# Result:
# ❌ Deleted anonymous volume containing important data!
# ✅ Named volumes are safe (not removed by prune)
```

### 5. Migration Complexity

```bash
# Named volume: Easy migration
docker run --rm \
  -v meepleai_postgres_data:/source:ro \
  -v new_server_postgres_data:/target \
  alpine sh -c "cp -a /source/. /target/"

# Anonymous volume: Must find correct hash first!
# 1. Inspect container to find volume mount
# 2. Match hash to volume
# 3. Hope it's the right one
# 4. Pray during migration
```

## Detection & Identification

### Find Anonymous Volumes

```bash
# List all volumes
docker volume ls

# Filter by name pattern (anonymous = long hash)
docker volume ls --format "{{.Name}}" | grep -E "^[a-f0-9]{64}$"

# Count anonymous volumes
docker volume ls --format "{{.Name}}" | grep -E "^[a-f0-9]{64}$" | wc -l
```

### Identify Volume Owner

```bash
# Method 1: Inspect volume
docker volume inspect a7f3d8c9b1e2... | jq '.[0].Labels'

# Output (if labeled):
{
  "com.docker.compose.project": "meepleai",
  "com.docker.compose.volume": "postgres_data"
}

# Method 2: Find which container uses it
for container in $(docker ps -a -q); do
  docker inspect $container | jq -r '.[] | .Name + " uses: " + (.Mounts[].Name // "none")'
done | grep "a7f3d8c9"
```

### Find Dangling Volumes

```bash
# Volumes not attached to any container
docker volume ls --filter "dangling=true"

# Count
docker volume ls -q --filter "dangling=true" | wc -l
```

## Prevention Strategies

### 1. Always Declare Named Volumes

**✅ Correct Pattern**:

```yaml
# docker-compose.yml
name: meepleai  # Project name for consistent prefixes

services:
  postgres:
    image: postgres:16
    volumes:
      - postgres_data:/var/lib/postgresql/data  # Named volume

  redis:
    image: redis:7
    volumes:
      - redis_data:/data  # Named volume

volumes:
  postgres_data:  # Explicit declaration
  redis_data:
```

**Result**:
```bash
docker volume ls
# local     meepleai_postgres_data  ✅
# local     meepleai_redis_data     ✅
```

### 2. Check Base Image VOLUME Directives

```bash
# Inspect base image
docker inspect postgres:16 | jq '.[0].Config.Volumes'

# Output:
{
  "/var/lib/postgresql/data": {}  # This creates anonymous volume if not overridden!
}
```

**Solution**: Always override in docker-compose.yml

### 3. Use Volume Labels

```yaml
volumes:
  postgres_data:
    labels:
      com.meepleai.backup: "critical"
      com.meepleai.service: "postgres"
      com.meepleai.purpose: "database-storage"
```

### 4. Document Volume Purpose

```yaml
volumes:
  # CRITICAL - Database storage (hourly backups)
  postgres_data:

  # CRITICAL - Vector database (daily backups)
  qdrant_data:

  # HIGH - User uploaded PDFs (daily backups)
  pdf_uploads:

  # LOW - Cache only (no backup needed)
  redis_data:
```

## Cleanup Procedures

### Safe Cleanup (Recommended)

```bash
# 1. List anonymous volumes
docker volume ls --format "{{.Name}}" | grep -E "^[a-f0-9]{64}$"

# 2. Check if dangling (not used by any container)
docker volume ls -q --filter "dangling=true"

# 3. Remove dangling only (safe)
docker volume prune -f

# Output:
# Deleted Volumes:
# a7f3d8c9b1e2f48a5c3d1e9f0b4a6c8d2
# 2c1b8a9d3e4f5a6b7c8d9e0f1a2b3c4d
# Total reclaimed space: 5.2GB
```

### Aggressive Cleanup (DANGEROUS!)

```bash
# Remove ALL unused volumes (named + anonymous)
docker volume prune -a -f

# WARNING: This can delete:
# - Anonymous volumes (intended)
# - Named volumes not currently attached (DANGEROUS!)
# - Volumes from stopped containers

# Safer alternative: Remove only anonymous
for vol in $(docker volume ls -q --filter "dangling=true"); do
  # Verify it's anonymous (64 char hex)
  if [[ $vol =~ ^[a-f0-9]{64}$ ]]; then
    echo "Removing anonymous volume: $vol"
    docker volume rm $vol
  fi
done
```

### Scheduled Cleanup

```bash
#!/bin/bash
# /opt/meepleai/scripts/cleanup-anonymous-volumes.sh

echo "🧹 Cleaning up anonymous Docker volumes..."

# Count before
BEFORE=$(docker volume ls -q | wc -l)

# Remove dangling anonymous volumes
docker volume prune -f

# Count after
AFTER=$(docker volume ls -q | wc -l)
REMOVED=$((BEFORE - AFTER))

echo "✅ Removed $REMOVED dangling volumes"
echo "📊 Remaining volumes: $AFTER"

# Log cleanup
echo "$(date): Removed $REMOVED volumes" >> /var/log/docker-cleanup.log
```

**Cron Schedule**:
```bash
# Weekly cleanup (Sunday 2 AM)
0 2 * * 0 /opt/meepleai/scripts/cleanup-anonymous-volumes.sh
```

**Problem**:

```yaml
# BAD: No volume override
services:
  postgres:
    image: postgres:16
```

```bash
# Result: Anonymous volume created

# After docker compose down -v
# ❌ Data lost!
```

**Solution**:
```yaml
# GOOD: Named volume override
services:
  postgres:
    image: postgres:16
    volumes:
      - postgres_data:/var/lib/postgresql/data

  postgres_data:
```


### Case 2: Orphaned Volumes After Rebuild

**Scenario**: Developer rebuilds containers frequently

```bash
# Initial state
docker volume ls | wc -l
# 12 volumes

# After 10 rebuilds without cleaning up
docker volume ls | wc -l
# 47 volumes (35 orphaned anonymous!)

# Disk usage
docker system df -v | grep Volumes
# Volumes space usage: 15.2GB (10GB wasted on orphaned volumes)
```

**Solution**:
```bash
# Add to development workflow
docker compose down
docker volume prune -f  # Remove orphaned
docker compose up -d

# Or: Cleanup script
#!/bin/bash
# cleanup.sh
docker compose down
docker volume prune -f
docker image prune -f
docker compose up -d
```

### Case 3: Migration Nightmare

**Scenario**: Moving to new server, need to identify volumes

```bash

docker volume ls

# 2c1b8a9d...  # Database? Cache? Uploads?

# Manual investigation required

  echo "=== Volume: $vol ==="
```

## Detection & Identification

### Automated Detection Script

```bash
#!/bin/bash
# detect-anonymous-volumes.sh

echo "🔍 Scanning for anonymous volumes..."

# Find all volumes
ALL_VOLUMES=$(docker volume ls -q)
ANONYMOUS_COUNT=0
NAMED_COUNT=0

for vol in $ALL_VOLUMES; do
  # Check if hash pattern (64 hex chars)
  if [[ $vol =~ ^[a-f0-9]{64}$ ]]; then
    echo "❌ ANONYMOUS: $vol"

    # Try to find owner
    OWNER=$(docker volume inspect $vol | jq -r '.[0].Labels["com.docker.compose.project"] // "unknown"')
    echo "   Project: $OWNER"

    # Check if dangling
    DANGLING=$(docker volume ls -q --filter "name=$vol" --filter "dangling=true")
    if [ -n "$DANGLING" ]; then
      echo "   Status: DANGLING (not used)"
    else
      echo "   Status: IN USE"
    fi

    ((ANONYMOUS_COUNT++))
  else
    echo "✅ NAMED: $vol"
    ((NAMED_COUNT++))
  fi
done

echo ""
echo "📊 Summary:"
echo "   Named volumes: $NAMED_COUNT"
echo "   Anonymous volumes: $ANONYMOUS_COUNT"
echo "   Total: $((NAMED_COUNT + ANONYMOUS_COUNT))"

if [ $ANONYMOUS_COUNT -gt 0 ]; then
  echo ""
  echo "⚠️  WARNING: Found $ANONYMOUS_COUNT anonymous volumes"
  echo "   Run: docker volume prune -f (to remove dangling)"
fi
```

### Identify Volume Contents

```bash
# List files in anonymous volume (to identify purpose)
docker run --rm -v a7f3d8c9b1e2...:/data alpine ls -lR /data | head -20

# Check for database files
docker run --rm -v a7f3d8c9b1e2...:/data alpine find /data -name "*.db" -o -name "PG_VERSION"

# Check file count
docker run --rm -v a7f3d8c9b1e2...:/data alpine sh -c "find /data -type f | wc -l"

# Check size
docker volume inspect a7f3d8c9b1e2... | jq '.[0].Size'
```

## Prevention Strategies

### 1. Project Template

```yaml
# docker-compose.template.yml
# Use this as starting point for all new services

name: meepleai  # Consistent project name

services:
  example-service:
    image: example:latest
    volumes:
      - service_data:/path/to/data  # ✅ Always named

volumes:
  service_data:  # ✅ Always declared
    labels:
      com.meepleai.backup: "true|false"
      com.meepleai.service: "service-name"
      com.meepleai.criticality: "critical|high|medium|low"
```

### 2. Pre-Commit Hook

```bash
#!/bin/bash
# .git/hooks/pre-commit

# Check docker-compose.yml for anonymous volumes
if git diff --cached --name-only | grep -q "docker-compose"; then
  echo "🔍 Checking for anonymous volumes..."

  # Simple heuristic: volumes without volume declaration
  if grep -q "volumes:" infra/docker-compose.yml; then
    if ! grep -A 100 "^volumes:" infra/docker-compose.yml | grep -q "  [a-z_]*:"; then
      echo "❌ ERROR: Possible anonymous volume detected!"
      echo "   All volumes must be declared in volumes: section"
      exit 1
    fi
  fi
fi
```

### 3. CI/CD Check

```yaml
# .github/workflows/validate-docker-compose.yml
name: Validate Docker Compose

on: [pull_request]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Check for Anonymous Volumes
        run: |
          cd infra

          # Extract volume mounts
          MOUNTS=$(docker compose config | grep -A 1 "volumes:" | grep -v "^--" | grep -v "^volumes:")

          # Check each mount has corresponding declaration
          for mount in $MOUNTS; do
            VOL_NAME=$(echo $mount | cut -d: -f1 | xargs)
            if ! grep -q "^  $VOL_NAME:" docker-compose.yml; then
              echo "❌ Volume $VOL_NAME not declared in volumes section"
              exit 1
            fi
          done

          echo "✅ All volumes are properly declared"
```

### 4. Documentation Standard

```markdown
# When adding new service, ALWAYS document:

## Service: example-service

### Volumes
- `example_data` - Purpose: Service data storage
  - Path: `/path/to/data`
  - Backup: Daily (critical)
  - Size estimate: 5-10 GB
  - Retention: 30 days
```

## Cleanup Procedures

### Development Environment

```bash
# Safe cleanup (removes only dangling)
docker compose down
docker volume prune -f
docker compose up -d

# Verify no data loss
docker compose ps
curl http://localhost:8080/health
```

### Production Environment

```bash
# NEVER run docker volume prune on production!
# Instead: Identify and remove specific volumes

# 1. List all volumes
docker volume ls

# 2. Identify anonymous + dangling
docker volume ls -q --filter "dangling=true" | grep -E "^[a-f0-9]{64}$"

# 3. Backup first (just in case)
for vol in $(docker volume ls -q --filter "dangling=true"); do
  docker run --rm -v $vol:/src:ro -v /backups/orphaned:/dst alpine \
    tar czf /dst/$vol.tar.gz -C /src .
done

# 4. Remove after verification
docker volume prune -f

# 5. Monitor for issues
tail -f /var/log/syslog | grep docker
```

### Migration Cleanup

```bash
# After migrating to new server
# OLD SERVER:

# 1. Stop all containers
docker compose down

# 2. Backup named volumes
for vol in $(docker volume ls --filter label=com.docker.compose.project=meepleai -q); do
  docker run --rm -v $vol:/src:ro -v /backups:/dst alpine \
    tar czf /dst/$vol.tar.gz -C /src .
done

# 3. Transfer to new server
rsync -avz /backups/ user@new-server:/backups/

# NEW SERVER:
# 4. Restore volumes (see docker-volume-management.md)
# 5. Start services
# 6. OLD SERVER: Cleanup
docker system prune -a -f --volumes
```

## Best Practices Summary

### ✅ DO

1. **Always use named volumes** in docker-compose.yml
2. **Set project name** with `name:` directive
3. **Document volume purpose** with comments
4. **Label volumes** with metadata (backup, service, criticality)
5. **Regular cleanup** of dangling volumes (weekly in dev)
6. **Verify volumes** before and after cleanup
7. **Backup before** major operations

### ❌ DON'T

1. **Never rely on anonymous volumes** for persistent data
2. **Never run `docker volume prune`** on production without verification
3. **Never use `docker compose down -v`** without backup
4. **Never assume** a hash is safe to delete without checking
5. **Never skip** volume declaration in docker-compose.yml
6. **Never ignore** dangling volume warnings
